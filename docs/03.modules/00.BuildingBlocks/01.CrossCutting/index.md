# نمای کلی قابلیت‌های عمومی و بین‌برشی

## هدف این سند

این سند خانواده «قابلیت‌های عمومی و بین‌برشی» را در دسته BuildingBlocks تعریف می‌کند.

نقش این سند:

* تعریف دقیق این خانواده
* مشخص کردن مرز آن با سایر خانواده‌ها
* معرفی اعضای این خانواده
* ثبت تصمیم‌های مشترک
* ایجاد ورودی برای مستندات هر ماژول

این سند وارد جزئیات تفصیلی هر ماژول نمی‌شود.

---

## تعریف این خانواده

این خانواده شامل capabilityهایی است که:

* در بخش‌های مختلف سیستم استفاده می‌شوند
* وابسته به یک لایه خاص نیستند
* وابسته به یک دامنه خاص نیستند
* در چندین پروژه و سناریو قابل بازاستفاده هستند

این قابلیت‌ها معمولاً:

* زیرساختی هستند
* بین‌برشی هستند
* در چند نقطه از سیستم مصرف می‌شوند

---

## چرا این خانواده در BuildingBlocks قرار می‌گیرد

اعضای این خانواده:

* به‌صورت مستقل قابل استفاده‌اند
* خودبسنده هستند
* وابستگی به یک flow یا الگوی خاص ندارند
* می‌توانند بدون داشتن کل سیستم استفاده شوند

بنابراین:
این‌ها capability خام هستند، نه الگوی سطح اپلیکیشن.

---

## مرز این خانواده با سایر خانواده‌ها

### تفاوت با Runtime و Registration

* Runtime بیشتر مربوط به کشف، wiring و startup است
* CrossCutting مربوط به capabilityهای مصرفی در طول اجرای سیستم است

مثال:

* `DependencyInjection` → CrossCutting
* `SoftwarePartDetector` → Runtime

---

### تفاوت با Data و Persistence

* Data مربوط به ماندگاری و دیتابیس است
* CrossCutting مستقل از storage است

---

### تفاوت با Domain و Application Primitives

* Domain primitives مفاهیم مدل‌سازی هستند
* CrossCutting ابزارها و capabilityهای عمومی هستند

---

### تفاوت با Messaging

* Messaging مربوط به رویداد و پیام است
* CrossCutting ابزارهای عمومی هستند که ممکن است در messaging هم استفاده شوند

---

## اعضای این خانواده

در وضعیت فعلی، این خانواده شامل این ماژول‌هاست:

* `Object Mapper`
* `Translations`
* `Serializer`
* `Caching`
* `DependencyInjection`
* `Sinks`

این فهرست اولیه است و در ادامه ممکن است توسعه پیدا کند.

## توضیح کوتاه اعضا

### Object Mapper

قابلیتی برای نگاشت بین آبجکت‌ها که تصمیم استفاده از ابزار mapping را پشت یک مرز انتزاعی محصور می‌کند.

نام محصولی این capability در زمین X، `Morpher` است و نام فنی آن `Object Mapper` است.

برای جزئیات بیشتر:

* `object-mappers.md`

---

### Translations

قابلیتی برای مدیریت ترجمه متن‌ها.

نام محصولی این capability در زمین X، `Parrot` است.
این capability با مدل «انتزاع + provider» طراحی می‌شود، از چند فراهم‌کننده داده با ترتیب مشخص پشتیبانی می‌کند، داده‌ها را در حافظه نگه می‌دارد و امکان refresh بدون restart را فراهم می‌کند.
provider اولیه این capability، نسخه SQL Server است.

برای جزئیات بیشتر:

* `Translator.md`

---

### Serializer

قابلیتی برای serialization و deserialization داده که در فاز فعلی روی JSON متمرکز است.

نام محصولی این capability در زمین X، `Prism` است.
این capability با مدل «انتزاع + provider» طراحی می‌شود و providerهای اولیه آن شامل نسخه مایکروسافتی و نسخه نیوتن‌سافتی هستند.

برای جزئیات بیشتر:

* `serializer.md`

---

### Caching

قابلیتی برای نگهداری موقت داده‌ها با هدف بهبود performance و کاهش وابستگی مستقیم مصرف‌کننده‌ها به backendهای مختلف cache.

نام محصولی این capability در زمین X، `StashX` است.
این capability با مدل **replacement-based provider capability** طراحی شده و یک contract مصرفی کوچک و پایدار با نام `IStashX` ارائه می‌کند.
providerهای اولیه آن شامل نسخه InMemory، Redis و SqlServer هستند.
در provider مربوط به SqlServer، setup storage به‌صورت explicit و opt-in پشتیبانی می‌شود.

برای جزئیات بیشتر:

* `caching.md`

---

### DependencyInjection

قابلیتی برای مدیریت وابستگی‌ها و wiring اجزا.

برای جزئیات بیشتر:

* `dependency-injection.md`

### Sinks

قابلیتی برای ارسال داده‌ها به خروجی‌های مختلف.

برای جزئیات بیشتر:

## تصمیم‌های مشترک این خانواده

در وضعیت فعلی، این تصمیم‌ها برای کل خانواده مبنا هستند:

* این capabilityها باید مستقل از لایه‌های خاص باشند
* باید بدون نیاز به کل سیستم قابل استفاده باشند
* setup آن‌ها باید ساده و قابل درک باشد
* در صورت نیاز، helperها و extensionها در کنار آن‌ها ارائه می‌شوند
* وابستگی به تکنولوژی مانع قرارگیری در این خانواده نیست
* اگر نیاز واقعی به مرز انتزاعی وجود داشته باشد، capability می‌تواند با مدل «انتزاع + provider» طراحی شود

---

## الگوی طراحی در این خانواده

در این خانواده معمولاً یکی از این دو الگو استفاده می‌شود:

### حالت ۱: استفاده از انتزاع موجود

اگر بستر .NET یا اکوسیستم انتزاع مناسبی داشته باشد:

* از همان استفاده می‌شود
* فقط setup و integration ساده‌سازی می‌شود

### حالت ۲: تعریف انتزاع اختصاصی

اگر انتزاع مناسبی وجود نداشته باشد یا نیاز واقعی به مرز capability وجود داشته باشد:

* یک قرارداد برای capability تعریف می‌شود
* یک یا چند provider پشت آن قرار می‌گیرند
* ابزار بیرونی از سطح مصرف‌کننده پنهان می‌ماند

Object Mapper نمونه فعلی این الگو است، Serializer دومین capability مرجع این خانواده است که با همین رویکرد طراحی شد، Translator با نام محصولی `Parrot` نمونه مرجع الگوی core-orchestrated در این خانواده است، و Caching با نام محصولی `StashX` نمونه capability provider-based از نوع replacement-based در این خانواده است.

---

## نسبت با سایر دسته‌ها

### نسبت با ApplicationPatterns

ApplicationPatterns ممکن است از این capabilityها استفاده کنند، اما این خانواده خودش pattern ارائه نمی‌دهد.

---

### نسبت با Integrations

در بعضی موارد، یک Integration ممکن است از این capabilityها استفاده کند، اما این خانواده خودش مسئول اتصال به بیرون نیست.
