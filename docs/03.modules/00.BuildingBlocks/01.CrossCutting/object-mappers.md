این سند ماژول Object Mapper را در زمین X تعریف می‌کند.

نقش این سند:
- تعریف مسئله‌ای که Object Mapper حل می‌کند
- توضیح چرایی وجود این capability در زمین X
- مشخص کردن جایگاه آن در taxonomy پروژه
- ثبت ساختار فعلی ماژول
- ثبت تصمیم‌های طراحی
- روشن کردن نسبت آن با providerها و مصرف‌کننده‌ها
- تثبیت Object Mapper به‌عنوان یکی از نمونه‌های مرجع برای طراحی capabilityهای عمومی و بین‌برشی

این سند وارد جزئیات providerهای آینده، naming packageها یا تصمیم‌های انتشار نمی‌شود.

---

## تعریف ماژول

Object Mapper یک BuildingBlock از خانواده قابلیت‌های عمومی و بین‌برشی است که یک مرز انتزاعی برای نگاشت بین آبجکت‌ها ارائه می‌کند.

هدف این capability فقط انجام mapping نیست.  
هدف اصلی آن این است که تصمیم استفاده از یک ابزار یا روش مشخص برای mapping در سطح پروژه پخش نشود و مصرف‌کننده‌ها به‌جای وابستگی مستقیم به ابزار، به یک قرارداد پایدار وابسته شوند.

در سطح محصول، نام این capability «Object Mapper» است.

در ساختار مستندات، این capability در فایل `object-mappers.md` ثبت می‌شود.

---

## مسئله‌ای که حل می‌کند

در تقریباً همه پروژه‌ها نیاز به نگاشت بین آبجکت‌ها وجود دارد.

نمونه‌های رایج:
- نگاشت entity به DTO
- نگاشت مدل ورودی به مدل دامنه
- نگاشت مدل دامنه به مدل خروجی
- نگاشت بین مدل‌های میانی

در بسیاری از پروژه‌ها این تصمیم به‌صورت مستقیم در سطح کد مصرف‌کننده پخش می‌شود.  
مثلاً:
- همه‌جا از یک کتابخانه مشخص mapping استفاده می‌شود
- یا همه‌جا mapping دستی با الگوهای ناهماهنگ انجام می‌شود

این رویکرد چند مشکل ایجاد می‌کند:
- تصمیم فنی انتخاب ابزار در سراسر پروژه پخش می‌شود
- تغییر provider یا روش پیاده‌سازی پرهزینه می‌شود
- تغییر API ابزار بیرونی می‌تواند خرابی را در بخش‌های زیادی از پروژه منتشر کند
- استفاده نادرست یا غیریکدست از ابزارهای بیرونی افزایش پیدا می‌کند
- کنترل معماری روی مرز capability از بین می‌رود

Object Mapper برای جلوگیری از این مسائل ایجاد می‌شود.

---

## چرایی وجود Object Mapper در زمین X

چرایی وجود این capability فقط «قابل جایگزینی بودن» نیست.

این capability برای این در زمین X وجود دارد که:

- تصمیم استفاده از ابزار mapping در سطح پروژه محصور شود
- تغییر provider باعث تغییرات گسترده در مصرف‌کننده‌ها نشود
- تغییر API ابزار بیرونی در سطح سیستم منتشر نشود
- استفاده صحیح و یکدست از ابزارهای mapping در یک نقطه متمرکز شود
- وابستگی مصرف‌کننده‌ها به کتابخانه‌های بیرونی حذف شود
- امکان ارائه providerهای مختلف برای یک capability مشترک فراهم شود

به‌عبارت دیگر، Object Mapper یک capability مرزی است که هم از پروژه در برابر نشت تصمیم ابزار محافظت می‌کند و هم از پروژه در برابر تغییرات بیرونی.

---

## مثال مسئله

اگر مصرف‌کننده‌ها مستقیماً به یک ابزار بیرونی مانند AutoMapper وابسته شوند، این وابستگی در سراسر پروژه منتشر می‌شود.

در این وضعیت:
- API آن ابزار وارد لایه‌های مختلف سیستم می‌شود
- تغییر provider نیازمند بازبینی بخش‌های زیادی از کد می‌شود
- تغییرات نسخه‌ای ابزار بیرونی می‌تواند مصرف‌کننده‌ها را بشکند
- کیفیت استفاده از ابزار به دانش تک‌تک افراد تیم وابسته می‌شود

اما اگر این ابزار پشت یک انتزاع محصور شود:
- مصرف‌کننده فقط به قرارداد داخلی capability وابسته می‌شود
- provider را می‌توان عوض کرد
- تغییرات ابزار بیرونی درون capability جذب می‌شود
- استاندارد استفاده صحیح از ابزار در خود capability پیاده‌سازی می‌شود

## جایگاه در taxonomy

Object Mapper در دسته `BuildingBlocks` و در خانواده `01.CrossCutting` قرار می‌گیرد.

دلیل این جایگاه:
- یک capability مستقل و قابل بازاستفاده است
- به دامنه خاصی وابسته نیست
- به لایه خاصی وابسته نیست
- در چندین نقطه از سیستم قابل استفاده است
- الگوی سطح اپلیکیشن ارائه نمی‌دهد
- مسئله اصلی آن اتصال به boundary بیرونی نیست
- می‌تواند بدون داشتن کل سیستم به‌صورت مستقل مصرف شود

بنابراین Object Mapper یک capability عمومی و بین‌برشی است، نه یک ApplicationPattern و نه یک Integration.

## ساختار فعلی ماژول

در وضعیت فعلی، Object Mapper از این بخش‌ها تشکیل شده است:

### 1. لایه انتزاع
این بخش قرارداد اصلی capability را تعریف می‌کند.

در وضعیت فعلی، قرارداد اصلی این ماژول:
- `IMapperAdapter`

این قرارداد مرز اصلی مصرف capability است.

### 2. لایه provider
این بخش پیاده‌سازی‌های مشخص capability را ارائه می‌کند.

در وضعیت فعلی:
- provider مبتنی بر AutoMapper وجود دارد

این provider مسئول پیاده‌سازی قرارداد اصلی Object Mapper است.

### 3. لایه registration و setup
هر provider مسئول registration و setup خودش است.

در وضعیت فعلی، provider AutoMapper extension methodهای لازم برای registration را درون خودش نگه می‌دارد.

### 4. لایه sample
یک sample اجرایی برای نمایش استفاده واقعی از capability وجود دارد.

هدف sample:
- نمایش registration
- نمایش inject شدن قرارداد
- نمایش استفاده واقعی از capability
- نمایش scan شدن assemblyهای مرتبط
- نمایش mapping در یک سناریوی ساده

## مدل طراحی

Object Mapper با مدل provider-based طراحی می‌شود.

در این مدل:
- مصرف‌کننده به قرارداد capability وابسته می‌شود
- providerها قرارداد را پیاده‌سازی می‌کنند
- registration در provider نگه داشته می‌شود
- ابزار بیرونی از سطح مصرف‌کننده مخفی می‌ماند

این مدل باعث می‌شود:
- providerهای مختلف قابل اضافه شدن باشند
- تغییر provider به مصرف‌کننده نشت نکند
- setup استاندارد و قابل کنترل باقی بماند
- دانش استفاده صحیح از ابزار درون capability متمرکز شود

---

## نقش انتزاع در این capability

در زمین X، ایجاد انتزاع یک قانون پیش‌فرض برای همه‌چیز نیست.  
اما در مورد Object Mapper، وجود انتزاع یک نیاز واقعی است.

دلیل این نیاز:
- جلوگیری از نشت ابزار بیرونی به مصرف‌کننده
- جلوگیری از انتشار تغییرات API ابزار بیرونی
- حفظ مرز capability
- امکان جایگزینی provider
- یکدست‌سازی نحوه استفاده

بنابراین انتزاع در Object Mapper یک انتزاع مصنوعی یا تزئینی نیست، بلکه یک مرز معماری ضروری است.

---

## پیاده‌سازی فعلی

در وضعیت فعلی، provider موجود برای Object Mapper مبتنی بر AutoMapper است.

این provider:
- قرارداد اصلی capability را پیاده‌سازی می‌کند
- setup و registration خودش را نگه می‌دارد
- امکان scan کردن assemblyهای مورد نیاز را فراهم می‌کند
- تنظیمات اضافی provider را از طریق options می‌پذیرد

این پیاده‌سازی فعلاً provider مرجع Object Mapper است، اما provider نهایی و انحصاری این capability در نظر گرفته نمی‌شود.

---

## registration و setup

یکی از تصمیم‌های طراحی در این capability این است که registration در خود provider نگه داشته شود.

این تصمیم باعث می‌شود:
- setup پراکنده نشود
- discoverability بهتر شود
- مسئولیت registration روشن بماند
- اضافه شدن provider جدید ساده‌تر شود

الگوی فعلی registration بر پایه extension method است.

در صورت نیاز به providerهای دیگر، هر provider باید registration خودش را در درون خودش ارائه کند.

## تصمیم‌های طراحی

### محصور کردن تصمیم ابزار
تصمیم انتخاب ابزار mapping نباید در سراسر پروژه پخش شود.

### محافظت در برابر تغییرات بیرونی
تغییر API ابزار بیرونی نباید باعث انتشار خرابی در مصرف‌کننده‌ها شود.

### تمرکز دانش استفاده صحیح
نحوه درست استفاده از ابزار mapping باید در خود capability متمرکز شود، نه در کد مصرف‌کننده‌ها.

### جداسازی قرارداد از provider
مصرف‌کننده به قرارداد وابسته می‌شود و provider پشت آن قرار می‌گیرد.

### مالکیت registration در provider
هر provider مسئول registration خودش است.

### پشتیبانی از توسعه آینده
طراحی باید امکان اضافه شدن providerهای دیگر را حفظ کند.

## نسبت با سایر ماژول‌ها

### نسبت با Dependency Injection
Object Mapper برای registration و تزریق از سازوکار DI استفاده می‌کند، اما خودش ماژول Dependency Injection نیست.

### نسبت با Serializers
Object Mapper مسئول نگاشت بین آبجکت‌ها است، نه تبدیل داده به فرمت متنی یا باینری.

### نسبت با Foundations و Applications
Foundationها و Applicationها می‌توانند مصرف‌کننده Object Mapper باشند، اما Object Mapper نباید بر اساس نیاز یک Foundation یا Application خاص طراحی شود.
