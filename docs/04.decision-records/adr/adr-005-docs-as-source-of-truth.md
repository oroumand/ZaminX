# ADR 005: docs مرجع اصلی تصمیم‌های محصولی و معماری است

## وضعیت
پذیرفته‌شده

---

## زمینه

در پروژه‌هایی که هم READMEهای محلی دارند و هم مستندات مرکزی، اگر مرجع اصلی مشخص نشود، اطلاعات بین چند محل پخش می‌شود و به‌مرور ناسازگاری ایجاد می‌شود.

در زمین X هم قرار است:
- docs لایه مرکزی مستندسازی باشد
- README کنار ماژول‌ها برای ورود سریع و اطلاعات محلی استفاده شود

اگر مرجع اصلی روشن نباشد:
- تصمیم‌های معماری در چند جا تکرار می‌شوند
- READMEها به اسناد معماری تبدیل می‌شوند
- docs اعتبار خود را از دست می‌دهد

---

## تصمیم

در زمین X، بخش `docs` مرجع اصلی تصمیم‌های محصولی، معماری، taxonomy، نقشه ماژول‌ها و وضعیت پروژه است.

READMEهای محلی:
- نقش محلی و فنی دارند
- برای ورود سریع، معرفی package و استفاده کوتاه نگه داشته می‌شوند
- جایگزین docs مرکزی نیستند

---

## دلایل تصمیم

این تصمیم برای این گرفته می‌شود که:
- source of truth روشن باشد
- تکرار مستندات کاهش پیدا کند
- مرز اسناد مرکزی و محلی مشخص شود
- consistency مستندات حفظ شود

---

## پیامدها

### پیامدهای مثبت
- محل رجوع برای تصمیم‌های اصلی روشن می‌شود
- READMEها سبک‌تر و مفیدتر می‌مانند
- docs نقش مرکزی خودش را حفظ می‌کند

### هزینه‌ها
- باید مراقب تکرار بین docs و READMEها بود
- نگهداری هم‌راستایی بین این دو لایه لازم است

---

## ارتباط با سایر اسناد

- `README.md`
- کل پوشه `docs`
