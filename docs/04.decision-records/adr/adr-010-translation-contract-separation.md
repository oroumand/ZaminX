# تفکیک قرارداد مصرفی از قرارداد تأمین داده در Translation

## وضعیت

پذیرفته‌شده

---

## زمینه

capability ترجمه در زمین X باید بتواند ترجمه‌ها را از sourceهای مختلف تأمین کند، بدون آن‌که API مصرفی به ابزار یا storage خاصی وابسته شود.

در این capability دو نقش متفاوت وجود دارد:

- مصرف ترجمه در لایه‌های application
- تأمین داده ترجمه از sourceهای مختلف مانند دیتابیس یا فایل

اگر این دو نقش در یک قرارداد واحد ترکیب شوند:

- جزئیات source داده به API مصرفی نشت می‌کند
- اضافه کردن یا جایگزینی providerها دشوار می‌شود
- استفاده هم‌زمان از چند source پیچیده یا غیرممکن می‌شود
- مرز capability با پیاده‌سازی مخدوش می‌شود

برای حفظ استقلال capability و پشتیبانی از مدل provider-based، نیاز به تفکیک این دو نقش وجود دارد.

---

## تصمیم

در capability Translation (Parrot)، قراردادها به دو بخش مستقل تفکیک می‌شوند:

- یک قرارداد مصرفی برای دریافت ترجمه (`ITranslator`)
- یک قرارداد جداگانه برای تأمین داده ترجمه (`ITranslationDataProvider`)

قرارداد مصرفی فقط مسئول ارائه API ترجمه به مصرف‌کننده است و هیچ وابستگی‌ای به source داده ندارد.

قرارداد تأمین داده فقط مسئول بارگذاری داده‌های ترجمه است و در API مصرفی ظاهر نمی‌شود.

## پیامدها

### مثبت

- API مصرفی ساده و مستقل از storage باقی می‌ماند
- providerها می‌توانند مستقل توسعه و جایگزین شوند
- امکان استفاده از چند source به‌صورت هم‌زمان فراهم می‌شود
- قابلیت تست‌پذیری capability افزایش می‌یابد

### منفی

- نیاز به orchestration داخلی برای ترکیب داده‌ها
- افزایش تعداد قراردادها در capability
- پیچیدگی بیشتر در طراحی merge و fallback
