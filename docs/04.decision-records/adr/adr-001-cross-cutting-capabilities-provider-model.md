# ADR 001: مدل انتزاع و provider برای capabilityهای عمومی و بین‌برشی

## وضعیت
پذیرفته‌شده

---

## زمینه

در زمین X تعدادی capability تکرارشونده و عمومی وجود دارند که در اغلب پروژه‌ها به آن‌ها نیاز می‌شود.

نمونه‌ها:
- Object Mapper
- Translations
- Serializers
- بعضی capabilityهای مشابه در خانواده قابلیت‌های عمومی و بین‌برشی

در این نوع capabilityها معمولاً یک یا چند پیاده‌سازی فنی یا ابزار بیرونی وجود دارد.  
برای مثال:
- در Object Mapper ممکن است از AutoMapper یا Mapster استفاده شود
- در Serializers ممکن است از یک serializer پیش‌فرض یا یک کتابخانه دیگر استفاده شود
- در Translation ممکن است داده‌ها از sourceهای مختلف تأمین شوند

اگر تصمیم استفاده از ابزار یا provider مستقیماً در سراسر پروژه پخش شود، مشکلات زیر ایجاد می‌شود:
- وابستگی مصرف‌کننده‌ها به ابزار بیرونی
- دشوار شدن جایگزینی provider
- انتشار تغییرات API ابزار بیرونی در سطح سیستم
- ناهماهنگی در شیوه استفاده از ابزارها
- از بین رفتن مرز capability

---

## تصمیم

برای capabilityهای عمومی و بین‌برشی که نیاز واقعی به مرز انتزاعی دارند، از مدل «انتزاع + provider» استفاده می‌کنیم.

در این مدل:
- مصرف‌کننده به قرارداد capability وابسته می‌شود، نه به ابزار بیرونی
- پیاده‌سازی‌های مختلف به‌صورت provider ارائه می‌شوند
- provider پشت قرارداد capability قرار می‌گیرد
- ابزار بیرونی نباید به API مصرفی capability نشت کند

---

## دلایل تصمیم

این تصمیم برای رسیدن به این اهداف گرفته می‌شود:
- محصور کردن تصمیم انتخاب ابزار در مرز capability
- جلوگیری از انتشار تغییر provider در سراسر پروژه
- جلوگیری از انتشار تغییرات API ابزارهای بیرونی
- متمرکز کردن دانش استفاده صحیح از ابزار
- حفظ مرز معماری capability

---

## پیامدها

### پیامدهای مثبت
- مصرف‌کننده‌ها به ابزار بیرونی وابسته نمی‌شوند
- providerهای مختلف قابل جایگزینی می‌شوند
- تغییرات ابزار بیرونی در درون capability جذب می‌شود
- استفاده از capability یکدست‌تر می‌شود
- طراحی capabilityهای مشابه می‌تواند بر پایه یک الگوی روشن انجام شود

### هزینه‌ها و محدودیت‌ها
- تعداد پروژه‌ها و packageها ممکن است بیشتر شود
- طراحی capability کمی پیچیده‌تر از اتصال مستقیم به ابزار بیرونی می‌شود
- برای capabilityهای ساده‌تر باید مراقب بود انتزاع غیرضروری ایجاد نشود

---

## دامنه تصمیم

این تصمیم یک قانون عمومی برای همه اجزا نیست.  
این مدل فقط زمانی استفاده می‌شود که نیاز واقعی به مرز انتزاعی وجود داشته باشد.

نشانه‌های این نیاز:
- احتمال وجود چند provider
- احتمال تغییر provider در آینده
- نیاز به جلوگیری از نشت ابزار بیرونی
- نیاز به کنترل شیوه استفاده از ابزار
- نیاز به جذب تغییرات API ابزار بیرونی در یک نقطه

اگر این شرایط وجود نداشته باشد، ایجاد انتزاع صرفاً به‌صورت پیش‌فرض توصیه نمی‌شود.
